# Anglesite Mac App ‚Äî Build Plan

**Status:** Active ‚Äî Phases 0‚Äì9 complete, Phase 10 in progress
**Companion design doc:** [`anglesite/docs/dev/mac-app-design.md`](../../anglesite/docs/dev/mac-app-design.md)
**Audience:** Contributors building the native macOS app in this repo.

This plan turns the high-level design into a concrete, phased implementation roadmap. Phases map to the v0 ‚Üí v2 milestones in ¬ß12 of the design doc.

## Phase 0 ‚Äî Repo + Xcode bootstrap

**Goal:** A buildable, signable, empty SwiftUI app committed to its own git repo.

1. `git init` in `Anglesite-app/`. Add `.gitignore` (Xcode, SwiftPM, DerivedData, `.DS_Store`, `node-runtime/`).
2. Create Xcode project: macOS App, SwiftUI lifecycle, Swift. *(Originally macOS 14; bumped to macOS 27+ after the Xcode 27 migration in #108.)*
   - Bundle id: `io.dwk.anglesite` (MAS) / `io.dwk.anglesite.devid` (DevID).
   - Capabilities: **off** sandbox for v0 (per ¬ß10), Hardened Runtime **on**, allow JIT + unsigned executable memory (Node needs both), allow DYLD env vars.
3. Add a top-level `README.md`, `LICENSE` (ISC to match plugin), `CLAUDE.md` (a short one ‚Äî points back to `anglesite/CLAUDE.md` for plugin context).
4. Add module structure inside the app target:
   - `AnglesiteApp/` ‚Äî SwiftUI views + app entry
   - `AnglesiteCore/` ‚Äî subprocess supervision, MCP client, edit pipeline
   - `AnglesiteBridge/` ‚Äî WKWebView script messages + JS injection
5. Set up CI (GitHub Actions) for `xcodebuild` build + unit tests on `macos-15`.
6. **Sign + notarize** dry run with a placeholder build to confirm Developer ID flow before any real code lands.

## Phase 1 ‚Äî Embedded Node runtime

This is the riskiest piece, so do it first.

1. ‚úÖ Decide on `node` vs `bun`. **Decided 2026-06-09: Node** ‚Äî Astro is officially supported, notarization is well-trodden, fewer surprises. (Removed from design-doc ¬ß13 in [paired plugin PR].)
2. ‚úÖ Vendor a Node.js macOS binary into `Resources/node-runtime/` via a build script (download + verify signature). *(Originally a lipo'd arm64/x86_64 universal binary; the x64 slice was dropped in #106 ‚Äî macOS 27 runs exclusively on Apple silicon.)*
3. ‚úÖ Re-sign the embedded `node` with the app's identity (notarization requires every Mach-O to be signed by the same team) (#4). Both targets now run `scripts/resign-node.sh` as a post-build phase: the **MAS target** with `Resources/node-runtime.entitlements` (app-sandbox/inherit + JIT, Task N), the **DevID `Anglesite` target** with `Resources/node-runtime-devid.entitlements` (the same JIT/unsigned-exec/disable-library-validation set *minus* the sandbox keys, since the DevID build isn't sandboxed). The script signs with `$EXPANDED_CODE_SIGN_IDENTITY` + `--options runtime` (Developer ID on a Release build with the cert; ad-hoc on Debug). Verified on an ad-hoc Debug build: the node carries the three entitlements with the `runtime` flag and `codesign --verify --deep --strict` reports the bundle valid + satisfies its Designated Requirement. The real Developer-ID-identity pass is exercised by the notarize dry run (#1), which needs the signing cert + `TEAM_ID`.
4. ‚úÖ Smoke test: spawn `node -e "console.log(1+1)"` from `NSTask`/`Process`, confirm it works in a notarized build. *(Ad-hoc-signed Debug confirmed; notarized confirmation deferred with step 3.)*
5. ‚úÖ Primed npm cache (#6): `scripts/vendor-npm-cache.sh` (prebuild phase, **default-on** ‚Äî set `ANGLESITE_BUILD_NPM_CACHE=0` to skip) installs the bundled plugin/template deps into a shared npm cache and writes a **gzipped** `Resources/npm-cache/cache.tar.gz` + a `version.txt` stamp. `AnglesiteCore/NodeModulesCache` extracts it on launch (`tar -xzf`) into `~/Library/Application Support/Anglesite/npm-cache/` (re-extracts only on a version bump ‚Äî `AnglesiteApp.prime()`) and exposes `npm install --prefer-offline --cache <path>` flags, which the New Site flow already passes (`SiteScaffolder`). **Size measured 2026-05-29:** ~768MB raw ‚Üí ~264MB gzipped ‚Äî over the original 100MB DMG-bloat budget, accepted as the price of offline first-launch (the alternative is the ~1min cold install the install-step degrades to gracefully). The script is idempotent (rebuilds only when `Resources/plugin/.bundled-from-commit` changes), so incremental builds skip the install.

## Phase 2 ‚Äî Plugin + site project plumbing

1. ‚úÖ Bundle a known-good copy of the Anglesite plugin in `Resources/plugin/` (copied from `../anglesite` at build time via the `Bundle Anglesite plugin` pre-build script in `project.yml`). Stamp the copy with the source commit for diagnostics.
2. ‚úÖ `SiteStore` (Swift actor): manages `~/Sites/<name>/` directories. Discovers existing projects (look for `anglesite.config.json`), persists the list in `~/Library/Application Support/Anglesite/sites.json`. *(Superseded by #242: `SiteStore` is now a recents registry of `.anglesite` packages ‚Äî `record`/`touch`, `recents.json` ‚Äî not a `~/Sites` scan. See CLAUDE.md "Site identity ‚Äî the `.anglesite` package".)*
3. ‚úÖ `ProjectValidator`: confirms a directory is an Anglesite project (`anglesite.config.json`, `astro.config.ts`, `keystatic.config.ts`). Returns which sentinels are missing so the UI can show partial-scaffold remediation.
4. ‚úÖ **Settings ‚Üí Advanced ‚Üí Plugin path** override (per ¬ß7) wired up early ‚Äî lets the plugin author point the app at `../anglesite` while iterating. `Sites root` override added alongside for development/testing.

## Phase 3 ‚Äî Subprocess supervisor

1. ‚úÖ `ProcessSupervisor` actor: spawns, restarts on crash (exponential backoff), streams stdout/stderr (via `LogCenter` + libdispatch `readabilityHandler`), handles graceful shutdown. Cancellable `waitForExit(_:)` so task groups can unwind without waiting for the real process exit. `shutdownAll(_:)` drains every supervised child ‚Äî wired to `AppDelegate.applicationShouldTerminate` so nothing outlives the app. `onRespawn` callback lets a wrapper re-establish session state after a supervised restart. App-wide instance is `ProcessSupervisor.shared`.
2. ‚úÖ `AstroDevServer`: wraps `astro dev` with a `parseReadyURL` regex (ANSI-stripped) and races URL match / unexpected exit / timeout in a single `withThrowingTaskGroup`. Once the `Local ‚Ä¶` line is spotted it is *probed over HTTP* (Astro logs the URL a beat before it serves) and only returned on a real response ‚Äî `ReadinessProbe` is injectable. Restart-on-crash is on by default (`maxAttempts: 3, baseBackoff: 0.5`); a watcher republishes `readyURL` after a restart picks a new port.
3. ‚úÖ `MCPClient`: spawns the server with `attachStdin: true`, speaks JSON-RPC 2.0 over stdio (NDJSON via LogCenter). v0 surface: `initialize`, `tools/list`, `tools/call`. Includes a `JSONValue` codec for typed but `Sendable` request/response shapes. Restart-on-crash is on by default; on respawn the client fails in-flight requests (`MCPError.reconnecting`) and re-runs the `initialize` handshake against the fresh process. *(The cross-repo `messages.mjs` schema-mirror CI check from issue #13 is deferred to Phase 5 ‚Äî Phase 3's surface is the standard MCP framing, with nothing app-specific to mirror yet.)*
4. ‚úÖ **Debug pane** behind `View ‚Üí Show Debug Pane` (`‚å•‚åòD`): live tail of all subprocess stdout/stderr from `LogCenter.shared`, with source/stream filter chips, free-text search, pause (freezes the view; the buffer keeps filling), auto-scroll, copy / save-to-file, and ring-buffer replay of pre-open history. Hidden in Release builds unless the user opts in (Settings ‚Üí Advanced ‚Üí Diagnostics) or holds ‚å• at launch; always shown in Debug builds. Filter/export logic lives in `LogCenter` extensions with unit coverage.

## Phase 4 ‚Äî WKWebView + edit overlay (v0 core)

1. ‚úÖ `PreviewView` (`NSViewRepresentable<WKWebView>`, AnglesiteApp): reloads `http://localhost:<port>` whenever the URL changes (incl. when a supervised dev-server restart rebinds a new port ‚Äî surfaced via `AstroDevServer.onReadyURLChange`). The testable core is `PreviewSession` (actor, AnglesiteCore): `start(siteID:siteDirectory:)` resolves how to run `astro dev` for a site (`node_modules/.bin/astro` via the vendored Node, or `.failed("dependencies not installed ‚Äî run npm install")`), drives an `AstroDevServer`, and exposes `State ‚àà {idle, starting, ready(url), failed(msg)}` plus an `observe()` change stream. `PreviewModel` (`@Observable`, AnglesiteApp) mirrors that state; `ContentView` is now a sidebar of discovered sites + a main pane that shows the live preview for the selected site (placeholder/error states otherwise) ‚Äî per-site *windows* are still Phase 9. `WebViewBridge.localDevConfiguration()` (no disk cache in Debug) + `applyLocalDevDefaults` (`isInspectable` in Debug, so ‚å•‚åòI devtools work); `Info.plist` gained `NSAllowsLocalNetworking` for the plain-http localhost load. *Note:* end-to-end "preview a real site" needs the site's `node_modules` (i.e. `npm install` having run ‚Äî see #6); until then the preview pane surfaces the "dependencies not installed" state and the tests use a `/bin/sh` fixture dev server.
2. ‚úÖ `WKScriptMessageHandler` for the `anglesite` namespace (#16). The JS ‚Üí native channel lives in `AnglesiteBridge`: `EditMessage.decode(from:)` strictly validates `{id, type, path, selector, op, value}` at the boundary (only `type == "anglesite:apply-edit"` is accepted; unknown types fail decode); `EditRouter` is the seam (`LoggingEditRouter` is the wired default ‚Äî logs to the Debug pane and replies `"Phase 5 lands the server side"`; `MCPApplyEditRouter` is the Phase-5-ready scaffold that calls `MCPClient.callTool(name: "anglesite:apply-edit", arguments: ‚Ä¶)` and maps the result). `AnglesiteScriptHandler` (`WKScriptMessageHandler`) decodes, routes, and replies via `evaluateJavaScript("window.anglesite?._handleReply?.(<EditReply>)")` on the originating WKWebView; the decode ‚Üí route ‚Üí reply construction is factored into a pure `handle(body:via:)` so it's unit-tested without `WKScriptMessage`. `WebViewBridge.localDevConfiguration(handler:)` registers the handler on the user-content controller; `PreviewView` wires a per-webview `AnglesiteScriptHandler(router: LoggingEditRouter())`.
3. ‚úÖ JS edit overlay (#17). Lives at `JS/edit-overlay/` (TS source) and builds to a single `Resources/edit-overlay/overlay.js` IIFE via esbuild ‚Äî `scripts/build-overlay.sh` runs as a prebuild phase (uses the vendored Node when present, system `npm` otherwise; idempotent on the `npm ci` step via `node_modules/.bin/esbuild` as the canary). Modules: `selector.ts` (pure `:nth-of-type` CSS path; first-cut strategy, #18 will finalize), `messages.ts` (post over `window.webkit.messageHandlers.anglesite`, await replies on `window.anglesite._handleReply`), `overlay.ts` (DOM behavior: hover outline / click ‚Üí `contentEditable=true` / blur ‚Üí post on change / drop image on `<img>` ‚Üí post `set-image` with base64). Injected via `WebViewBridge.makeOverlayUserScript(in:)` ‚Üí `WKUserScript` at `atDocumentEnd`, all frames; missing bundle is non-fatal (preview just loads without edit affordances). Tests via vitest + jsdom (19 in `JS/edit-overlay/test/` covering selector, messages, hover, click-to-edit, idempotent install) plus the Swift-side `WebViewBridgeTests` for the user-script loader. Drop-image behavior is implemented but not unit-tested (jsdom's DataTransfer/FileReader make it fiddly ‚Äî manual smoke instead).
4. ‚úÖ **Selector strategy: server-side resolution (#18, decided 2026-05-13).** The overlay sends a structured `ElementInfo` payload ‚Äî `{tag, id?, classes, nthChild, ancestors?, dataAnglesiteId?, dataTestId?, role?, ariaLabel?, textContent?}` matching the typedef in `anglesite/server/selector.mjs` ‚Äî and the plugin invokes `buildSelector(info)` server-side. One source of truth for the priority order (`data-anglesite-id` > `data-testid` > `#id` > role/aria > stable classes > `tag:nth-child`), no fork-prone duplicated JS. `JS/edit-overlay/src/selector.ts` exposes `elementInfoFor(element)`; ancestors are root-first and stop at `<body>`. The bridge stays a relay: `EditMessage.selector` is `JSONValue` (validated as an object at decode), forwarded as-is by `MCPApplyEditRouter` to the `anglesite:apply-edit` tool when Phase 5 wires it up.
5. ‚úÖ MCP client lifecycle + apply-edit round-trip wired (2026-05-22). `PreviewSession` (AnglesiteCore) now owns one `MCPClient` per site alongside its `AstroDevServer` ‚Äî spawned together in `start(siteID:siteDirectory:)`, drained together in `stop()`, both tracked by `ProcessSupervisor.shared` so app-quit drains every child. Spawn defaults: vendored Node + bundled plugin's `server/index.mjs` resolved via `PluginRuntime`, `ANGLESITE_PROJECT_ROOT` set to the site path, source tag `mcp:<siteName>`. Graceful failure ‚Äî if MCP can't spawn the session still reaches `.ready(url)`; only edit attempts fail (with the existing `EditReply.failed("MCP not running")` shape). `PreviewModel` builds an `MCPApplyEditRouter` wrapping a weak getter to `session.mcpClient` and exposes it as `editRouter`; `PreviewView`/`ContentView` thread it into `AnglesiteScriptHandler` (was `LoggingEditRouter`). End-to-end test in `AnglesiteBridgeTests/AppliesEditEndToEndTests` spawns the real bundled plugin against a tmp Astro-shaped fixture, drives `apply_edit` through a real `MCPClient`, asserts the file's bytes change ‚Äî `XCTSkip`s cleanly when the sibling plugin checkout or its `node_modules` aren't present (CI provides both via `actions/checkout` + `npm ci` + `ANGLESITE_PLUGIN_PATH`). Cost: +1 Node process per open site (~80‚Äì150 MB on top of the existing `astro dev` process).

## Phase 6 ‚Äî Deploy button (v0 finishing)

1. ‚úÖ `DeployCommand`: shells out to `wrangler deploy` from the site directory (#20). Cloudflare token from Keychain (Phase 7) or env.
2. ‚úÖ **Pre-deploy hook honored**: `PreDeployCheck` (AnglesiteCore) shells out to the site's `scripts/pre-deploy-check.ts --json` and parses the structured `ScanReport` (failures + warnings). `DeployCommand` runs the scan between the build step and `wrangler` resolution ‚Äî a `.blocked` outcome short-circuits to `Result.blocked(failures:, warnings:)` carrying `[PreDeployCheck.ScanFailure]` / `[PreDeployCheck.ScanWarning]`, which `ContentView` presents via `BlockedDeploySheetView` (a modal with no override, per the CLAUDE.md rule that the app cannot bypass plugin security hooks). The `npm run build` step now precedes the scan, so `dist/` is always fresh; build output streams under `deploy:<siteID>:build` and a non-zero build exit short-circuits to `.failed` before the scan runs.
3. ‚úÖ Output streamed to a transient drawer (#22). `DeployCommand` switched from `supervisor.run(...)` to `supervisor.launch(...)` so wrangler's stdout/stderr flow into `LogCenter` line-by-line; `DeployModel` (`@Observable`, AnglesiteApp) subscribes to `LogCenter` for the deploy's lifetime and exposes `phase`, `logLines`, and the drawer/sheet presentation flags. `DeployDrawerView` slides up from the bottom of the window with a live monospace log; on `.succeeded` it surfaces the deployed URL with **Copy URL** / **Open in browser** buttons; on `.failed` it keeps the log visible with a **Copy log** affordance. The drawer never auto-closes ‚Äî only the user's **Dismiss** action takes it down. URL extraction reads from a `LogCenter.snapshot()` filtered to the deploy source, with a 100ms grace period after `waitForExit` to let the supervisor's untracked pipe-pump Tasks land (the existing pump structure makes a fully synchronous drain impractical; revisit when the pump is restructured).

## Phase 7 ‚Äî Keychain + secrets

1. ‚úÖ `KeychainStore` (AnglesiteCore) wraps `SecItem` generic-password APIs under service `io.dwk.anglesite`, account `cloudflare-api-token`, accessibility `kSecAttrAccessibleAfterFirstUnlockThisDeviceOnly`. `DeployCommand.keychainTokenSource` is the new default `TokenSource` ‚Äî env var wins (so a developer shell with `CLOUDFLARE_API_TOKEN` exported still dominates), then the Keychain. `DeployModel.deploy(...)` does the same pre-check and parks the deploy behind a `CloudflareTokenPromptView` sheet when neither slot has a token; saving the token writes it to the Keychain and immediately fires the parked deploy. Settings ‚Üí Advanced ‚Üí Credentials gets a `CloudflareTokenRow` that surfaces "stored / absent" status and provides paste/save/clear without leaking the token through `@AppStorage`. The token never enters `LogCenter` ‚Äî it's passed via `Process.environment` (opaque to the supervisor's stdout/stderr pipe pumps). Tests cover `KeychainStore` round-trips against per-test scratch services (the `XCTSkip` path handles CI environments where the keychain refuses access).
2. ‚úÖ `gh` device-code flow stays in `gh` ‚Äî `GitHubAuthFlow` (AnglesiteCore) is an actor that spawns `gh auth login --git-protocol https --web --hostname github.com` via `ProcessSupervisor` (stdin attached so the flow can send Enter after the device-code prompt), parses the verification URL + one-time code out of gh's combined stdout/stderr stream (`PromptParser` strips ANSI, tolerates the wording variations gh has shipped, and waits until both pieces have been seen on whatever line they land), then yields `.devicePrompt(‚Ä¶) ‚Üí .authenticated | .failed(reason:)` via `AsyncStream<Event>`. The launcher is abstracted so tests drive the flow with fixture output. `GitHubAuthSheetView` (AnglesiteApp) consumes the stream, surfaces a Copy-able code + clickable URL + "Waiting for authentication‚Ä¶" spinner, opens the verification URL with `NSWorkspace.shared.open`, and reports the outcome to its parent. Settings ‚Üí Advanced ‚Üí Credentials gets a `GitHubAuthRow` that probes `gh auth status` on appear (handles "gh not installed" gracefully via a private PATH-walker) and offers a "Connect‚Ä¶" button that presents the sheet. The token never round-trips through Anglesite ‚Äî gh writes it directly into the user's keychain via `git-credential-osxkeychain`.

## Phase 8 ‚Äî v0.5 chat panel

1. ‚úÖ Per-site chat panel: `ChatHistoryStore` (AnglesiteCore) persists every user/assistant entry to `<site>/.anglesite/chat-history.jsonl` (append-only JSONL ‚Äî `tail -f` / `jq` friendly, corrupt-line tolerant). `ChatModel` (`@Observable`, `@MainActor`, AnglesiteApp) owns the per-site `ClaudeAgent`, accumulates the `AsyncStream<Event>` into typed `Message` values (with paired `ToolCall` records keyed by `toolUseID`), persists every terminal piece of content, and exposes `send(_:)` / `cancel()` / `resetConversation()`. `ChatView` (AnglesiteApp) is a right-hand panel toggled with ‚åòK from the preview-pane header: scrollable bubble list with native `AttributedString` markdown for assistant text, user-aligned blue bubbles, collapsible tool-call cards (input JSON / result / `isError` flag / spinner while pending), prompt input field with Send/Cancel buttons, and a usage telemetry line in the header (input/output tokens + cost + duration from `turnComplete`). Scroll-anchoring keeps the latest tokens visible as the response streams. **Permission sheets deferred:** Claude in `--print` mode doesn't pause for tool permission today ‚Äî the chat panel renders `toolUse` events as cards as the agent emits them; native permission sheets ride along with a follow-up that introduces a `--permission-prompt-tool` hook on `ClaudeAgent`.
2. ‚úÖ `ClaudeAgent` (AnglesiteCore) is an actor bound to a site directory; each `send(prompt:)` spawns `claude --print --output-format stream-json --verbose --plugin-dir <bundled> [--continue]` via `ProcessSupervisor` (source `claude:<siteID>`), parses the JSONL stream into typed `Event`s (`sessionStarted`, `assistantText`, `assistantThinking`, `toolUse`, `toolResult`, `turnComplete(usage:, costUSD:, durationMs:, stopReason:)`, `streamError`, `cancelled`, `processExited(code:)`), and yields them via `AsyncStream<Event>`. The parser is a pure stateless `StreamJSONParser.parse(line:)` so one assistant line carrying text + tool_use + thinking expands cleanly to three events. `--continue` is added on the second and later sends so claude resumes the conversation in the site directory; `resetSession()` clears that bit when the user wants a fresh chat. `cancel()` SIGTERMs the in-flight turn (the next send still resumes via `--continue`). The `Launcher` is an injectable abstraction ‚Äî production uses `makeSupervisorLauncher`, tests drive the actor with fixture JSONL. Permission-sheet handling for `toolUse` events is deferred to #25 (ChatView) ‚Äî the agent surfaces tool events as data; the chat panel decides the UI.
3. ‚úÖ Skill quick-action buttons live in `ChatView` above the input. `SkillRegistry` (AnglesiteCore) parses each bundled skill's `SKILL.md` frontmatter (tiny custom parser ‚Äî name + description, no YAML dep) and `quickActions(in:)` filters/orders the discovery to the curated v0.5 core set (`deploy`, `backup`, `check`, `import` ‚Äî hard-coded names in `quickActionNames`; labels and tooltips come from the plugin so they stay in sync as descriptions evolve). Clicking a button injects `/anglesite:<name>` into the model and submits ‚Äî same path as the user typing it. Icons are SF Symbols per name (paperplane / externaldrive / checkmark.shield / tray.and.arrow.down); unmapped names fall back to `sparkles`. Disabled while a turn is streaming. Falls through silently when the bundled plugin is missing.
4. ‚úÖ Sticky-note (annotation) routing into chat. The plugin's `annotations.mjs` is the source of truth ‚Äî annotations live at `<project>/annotations.json` and are exposed via the MCP tools `add_annotation` / `list_annotations` / `resolve_annotation`. `AnnotationFeed` (AnglesiteCore) is a `@Sendable` async closure typealias; `AnnotationFeedFactory.viaMCP(mcpClient:)` builds the production fetcher that calls `list_annotations` through the per-site `PreviewSession.mcpClient` (so chat reuses the same MCP server the edit overlay already drives ‚Äî no duplicate spawn). `ChatModel.loadAnnotations()` surfaces each unresolved annotation as a system message (`üìå <text> ‚Äî <sourceFile or path ‚Üí selector>`), idempotent across re-opens via a `surfacedAnnotationIDs` set so flipping back to a site doesn't re-post the same note. There's no standalone sticky-note UI to retire ‚Äî chat is the only surface that displays annotations from the start. A follow-up plugin change could add server-push notifications for new/resolved annotations; today the chat refreshes when the panel opens.
5. üü° Sparkle 2.x integration wired up. `Updater` (AnglesiteApp) wraps `SPUStandardUpdaterController` with `startingUpdater: true` so the automatic-check timer fires immediately at launch; bridges the controller's `canCheckForUpdates` into a `@Published` so the SwiftUI menu item disables itself while a check is in flight. "Check for Updates‚Ä¶" lives under the app menu directly below "About Anglesite". `project.yml` pulls Sparkle from `https://github.com/sparkle-project/Sparkle` pinned to `from: 2.6.0` (the only third-party Swift dep allowed at v0.5 per CLAUDE.md). `Info.plist` declares `SUFeedURL` (`https://anglesite.dev/appcast.xml`) and `SUPublicEDKey` (placeholder until keys are generated). **Remaining (manual setup):** run Sparkle's `generate_keys` once to produce the Ed25519 keypair (private key stays in the macOS Keychain), paste the public key into `Info.plist`, stand up the appcast URL, and fill in `scripts/release.sh` (the per-release pipeline is stubbed and documented in `docs/release.md`).

## Phase 9 ‚Äî v1 multi-site + drag-drop images

1. ‚úÖ Multi-window (#54): `WindowGroup(for: String.self)` keyed by `SiteStore.Site.id` (the path-derived stable string ‚Äî `SiteID.self` was the original sketch; `String` is what shipped because the value needs `Codable` for SwiftUI restoration). Each site opens in its own top-level window owning its `PreviewModel` / `DeployModel` / `ChatModel` as `@State`; dev-server lifecycle is bound to the window ‚Äî opening spins one up, closing tears it down (verified end-to-end via `scripts/create-smoke-fixture.sh`: two node children parented to the app, `:4321` Astro listener, fully drained by `ProcessSupervisor.shutdownAll` on close + quit). The `SitesLauncherView` (`Window("Sites", id: "sites")`) replaces the single-window sidebar; on first appearance of each app session it auto-opens the most-recently-used site via `AppSettings.lastOpenedSiteID` and dismisses itself. 2. ‚úÖ Health badge (#31). ## Containerization epic (#59) ‚Äî containerized dev server

**Tracking issue:** #59. Design: [`docs/specs/2026-05-30-cloudflare-sandbox-dev-server-design.md`](specs/2026-05-30-cloudflare-sandbox-dev-server-design.md).

The long-term architecture replaces the in-process Node subprocess with a container-backed dev server. **Apple Containerization is the primary runtime** ‚Äî it runs locally on macOS 27+ / Apple Silicon with near-native performance and no network dependency. **Cloudflare Sandbox is the fallback** for platforms and bundles where Apple Containerization is unavailable: the Mac App Store target (requires `com.apple.vm.networking`, which the App Store won't grant), iOS (no local container support), and any future non-Apple-Silicon targets. Both runtimes share a single OCI image and are fronted by the `SiteRuntime` protocol, so `PreviewModel` is unaware of which substrate is active. Runtime selection is automatic: prefer `LocalContainerSiteRuntime` when the host supports it; fall back to `RemoteSandboxSiteRuntime` otherwise.
