# CSP enforcement via `public/_headers` — design (#290)

**Date:** 2026-06-22
**Issue:** [#290](https://github.com/Anglesite/Anglesite-app/issues/290) — Bucket 3 follow-up: CSP enforcement via `public/_headers` for integration embeds
**Scope:** template-only (`Resources/Template/`); no `AnglesiteCore`/Swift changes

## Problem

Bucket 3 integration embeds (cal.com, Calendly, Stripe, Liberapay, GitHub Sponsors, giscus,
…) load third-party origins, but the scaffolded site emits **no Content-Security-Policy**.
There is nothing constraining which external origins a page may load scripts, frames, or
connections from.

The *data* needed to write a correct CSP already exists. The integration framework collects
each provider's allowed origins:

- Integration descriptors declare flat `cspDomains` per provider
  (`IntegrationCatalog.swift`: cal → `app.cal.com`; Stripe → `js.stripe.com`; giscus →
  `giscus.app`; etc.).
- The plan emits `.addCSP(domains)` operations.
- `IntegrationScaffolder` batches them into a single flat key in `.site-config`:
  `SCRIPT_ALLOW=app.cal.com,js.stripe.com,giscus.app`.

Nothing yet **reads** `SCRIPT_ALLOW`. This change adds the read/enforce end.

## Decisions

Three decisions were settled during brainstorming:

1. **Generation location — build time, not Swift scaffold time.**
   `public/_headers` is generated by a template build script that reads `.site-config`, not
   written by the Swift `IntegrationScaffolder`. Rationale: CLAUDE.md's "the filesystem is the
   source of truth" — a user (or the Claude Code CLI) hand-editing `.site-config` must also get
   correct headers. Build-time regeneration guarantees that; Swift-side writing would only
   update headers when the scaffolder ran. This also matches the sibling plugin's established
   `scripts/csp.ts` pattern.

2. **CSP precision — broad / flat.**
   Read the existing flat `SCRIPT_ALLOW` list and apply each domain to the directives embeds
   need (`script-src`, `frame-src`, `connect-src`, `img-src`). This honors the current
   provider-agnostic data model and keeps the change template-only. It is deliberately
   over-permissive (a booking domain is allowed to also frame/connect/load-images) in exchange
   for needing zero per-provider directive knowledge. A directive-aware refinement is explicitly
   out of scope and could be a later follow-up.

3. **Deploy gate — generate + validate.**
   `prebuild` regenerates `public/_headers`; **and** `pre-deploy-check.ts` asserts
   `dist/_headers` exists, contains a `Content-Security-Policy`, and that CSP covers every
   configured `SCRIPT_ALLOW` domain — erroring on a missing, empty, or incomplete CSP. This
   matches the app's "surface security failures, don't bypass" ethos
   (`pre-deploy-check` runs before every deploy).

## Data flow

```
.site-config  ──read──▶  scripts/csp.ts  ──write──▶  public/_headers
(SCRIPT_ALLOW=             (buildHeaders)            (Astro copies → dist/_headers)
 app.cal.com,
 js.stripe.com,                       pre-deploy-check.ts ──verify──▶ dist/_headers
 giscus.app)                          (errors if CSP missing / incomplete)
```

The generator runs with `cwd = Source/` (the scaffolded site), where both `.site-config` and
`public/` live.

## Components

### `Resources/Template/scripts/csp.ts` *(new)*

- `buildHeaders(configContent: string): string` — **pure** string→string. Parses
  `SCRIPT_ALLOW` from the given `.site-config` contents (via the existing
  `readConfigFromString` in `config.ts`), composes the full `_headers` file body, and returns
  it. No filesystem I/O — this is the unit-tested core.
- `main()` — thin CLI: reads `./.site-config` (tolerating absence), calls `buildHeaders`,
  writes `./public/_headers` (creating `public/` if needed). Overwrites wholesale each run
  (deterministic; no merge/drift logic).

### `Resources/Template/public/_headers` *(new, committed baseline)*

Secure-by-default headers committed so the file exists even before a build runs. `csp.ts`
regenerates it deterministically. Baseline content equals `buildHeaders("")` (no integrations).

### `Resources/Template/package.json`

- Add `"prebuild": "npx tsx scripts/csp.ts"` — npm runs this automatically before `build`, so
  `npm run build` can never emit a stale `_headers`.
- `check` also invokes the generator before scanning, so a local `npm run check` validates
  against freshly generated headers.

### `Resources/Template/scripts/pre-deploy-check.ts`

Add one check to the existing scan:

- `dist/_headers` exists → else `error`.
- It contains a `Content-Security-Policy:` line → else `error`.
- Every domain in `.site-config`'s `SCRIPT_ALLOW` appears in that CSP line → else `error`
  listing the missing domain(s).

When `.site-config` has no `SCRIPT_ALLOW`, only the "exists + has a CSP" assertions apply.

## CSP directive mapping (the "broad / flat" rule)

Baseline directives (secure-by-default, adapted from the sibling plugin's prior art):

```
default-src 'self';
script-src 'self' static.cloudflareinsights.com;
style-src 'self' 'unsafe-inline';
img-src 'self' data:;
font-src 'self';
connect-src 'self' cloudflareinsights.com;
frame-src 'self';
frame-ancestors 'none';
base-uri 'self';
form-action 'self'
```

Each domain in `SCRIPT_ALLOW` is appended to **`script-src`, `frame-src`, `connect-src`, and
`img-src`** — the four directives an integration embed needs (loader script, iframe,
XHR/beacon, images). Domains are deduplicated and emitted in stable (sorted) order for
reproducible output.

The full generated `public/_headers`:

```
/*
  X-Frame-Options: DENY
  X-Content-Type-Options: nosniff
  Referrer-Policy: strict-origin-when-cross-origin
  Permissions-Policy: camera=(), microphone=(), geolocation=(), payment=(), usb=(), interest-cohort=()
  Content-Security-Policy: <composed as above>
  Cache-Control: public, max-age=0, must-revalidate

/_astro/*
  Cache-Control: public, max-age=31536000, immutable
```

## Testing

- **`buildHeaders` unit tests** (node/tsx test under `scripts/`):
  - empty config → baseline only (no extra domains in any directive).
  - one integration domain → present in all four target directives, absent from others.
  - multiple domains, including a duplicate → deduped union, sorted.
  - malformed / blank-entry `SCRIPT_ALLOW` (e.g. `a,,b ,`) → blanks skipped, no crash.
- **pre-deploy-check tests**:
  - fixture `dist/` with no `_headers` → error.
  - `_headers` present but no CSP line → error.
  - CSP present but missing a configured `SCRIPT_ALLOW` domain → error naming the domain.
  - CSP covering all configured domains → passes.

## Error handling / edge cases

- No `.site-config`, or no `SCRIPT_ALLOW` key → baseline-only headers (still secure by default).
- Blank/whitespace domain entries in the list → skipped.
- `csp.ts` overwrites `public/_headers` wholesale every run — deterministic, no stale-merge risk.
- The dev server (`astro dev`) does not serve `_headers` (it is a production static-host
  concern), so `prebuild` on `build`/`check` is sufficient; no `predev` hook needed.

## Out of scope

- Directive-aware (per-provider) CSP precision — see decision 2.
- Porting the sibling plugin's full per-provider `csp.ts` builder suite
  (snipcart/shopify/paddle/lemon-squeezy/turnstile/tracking/podcast).
- Any `AnglesiteCore`/Swift change — the descriptor `cspDomains` → `SCRIPT_ALLOW` pipeline is
  already in place and unchanged.
